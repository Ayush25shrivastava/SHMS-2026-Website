SHMS-2026 — Contact form setup & e-mail configuration

The public site always shows the PHP contact form on contact.php (layout, CAPTCHA, CSRF). Submissions may be delivered either by PHP mail on the institute server or by a server-side POST to Google Apps Script (Sheet log plus MailApp notification with visitor Reply-To). The Sheet workflow does not use an embedded Google Form. If the Google Sheet and Apps Script are already configured and localhost testing is done, use the self-contained production guide contact-form-production-handoff.html (same folder as this file). Full-site hosting notes: DEPLOY-MNNIT.md at repository root. Below: extended reference for the institute web administrator, including optional Sheet-from-scratch steps.

Audience. These instructions are addressed to the designated institute web administrator (typically coordinated through the MNNIT Computer Centre). The SHMS‑2026 organising team has validated the application on localhost; production behaviour depends on Apache, PHP, and network settings on the institute Linux host. The institute’s public web pages do not publish internal Apache virtual-host templates; the steps below follow standard Apache 2.4 on Linux and must be reconciled with the exact allocation (document root, subdomain, PHP handler) supplied by the Computer Centre.

Part A — Deployment on institute Linux hosting (www.mnnit.ac.in)

Execute in order. Skip any step already satisfied by the host’s standard image.

  1. Obtain hosting parameters from the Computer Centre. Request: target URL (site root or subdirectory), filesystem path (DocumentRoot or alias target), supported PHP version, method of upload (SFTP, SCP, rsync, or internal pipeline), whether PHP runs as mod_php, php-fpm, or another SAPI, and TLS certificate practice for the vhost.
  2. Upload the deployable tree. Copy the project without development-only trees (e.g. omit node_modules). Preserve .htaccess, all root *.php, includes/, assets/, style.css, and api/. On the server, create includes/config.php from includes/config.sample.php; never publish repository secrets.
  3. Set ownership and permissions. Assign the document tree to the user/group required by Apache (often www-data on Debian/Ubuntu; institute images may differ). Typical practice: directories 755, files 644; restrict includes/config.php to the web user (e.g. 640) so credentials are not world-readable.
  4. Configure Apache for the allocated path. Ensure a <Directory> (or <VirtualHost>) block allows AllowOverride All (or equivalent) so the project’s .htaccess can set DirectoryIndex index.php and optional mod_rewrite rules. Example pattern (adjust paths and ServerName per allocation): 
    # Illustrative only — confirm with Computer Centre
<VirtualHost *:443>
    ServerName www.mnnit.ac.in
    DocumentRoot /var/www/shms2026
    <Directory /var/www/shms2026>
        Options -Indexes +FollowSymLinks
        AllowOverride All
        Require all granted
    </Directory>
    # SSL/TLS directives per institute standard
</VirtualHost>
    If the site lives in a subdirectory of an existing vhost, use an Alias (e.g. Alias /shms2026 /var/www/shms2026) with a matching <Directory> block; the application derives its base URL from SCRIPT_NAME and does not require rewriting every link.
  5. Enable required modules. mod_rewrite is recommended because .htaccess enables it for future rules. Run apache2ctl configtest (or httpd -t) before reload.
  6. Install and verify PHP extensions. At minimum: json, session. For Sheet webhook mode: php-curl (package name may be php8.x-curl per version). Ensure session.save_path in php.ini (or pool config for php-fpm) is writable by the PHP process.
  7. Allow outbound HTTPS. From the web server, TCP port 443 to script.google.com must succeed for form_backend sheet. Test from the server: curl -I https://script.google.com.
  8. Do not full-page cache contact.php. Disable caching for that URL in any reverse proxy, CDN, or mod_cache configuration so CSRF tokens and sessions remain valid.
  9. Reload Apache and verify. After sudo systemctl reload apache2 (or httpd), open the home page, an inner page, and contact.php; submit a test message if Sheet or PHP mail is configured.

Note. The Computer Centre’s software resources page references common server platforms (e.g. CentOS, Ubuntu); exact packages and service names depend on the OS image in use.

Part B — Configuration checklist for the hosting administrator (application layer). The Google Sheet, Apps Script project, and Web app deployment (including WEBHOOK_SECRET, SPREADSHEET_ID, and “Who has access: Anyone”) are maintained by the organising team. On the institute server, the administrator shall:
  1. Install the site files and includes/config.php as in Part A.
  2. For Sheet mode: set form_backend to sheet, and set sheet_webhook_url, sheet_webhook_secret, and mail_to exactly as documented in § E-mail messaging configuration.
  3. Confirm PHP cURL is loaded in the same SAPI as the browser (tools/php-curl-check.php via HTTPS).
  4. Ensure contact.php is not served from a static or full-page cache.

When form_backend is sheet, notification e-mail is sent by Google MailApp inside Apps Script, not by the institute MTA; SMTP entries in config.php do not affect that path. Use § E-mail messaging configuration for a precise breakdown.

What visitors see (sheet backend). The page uses neutral wording — for example: we receive your message at mail_to and reply to the address you provide. Technical storage details stay in this guide, not on the contact page.

Choose a backend

PHP mail / SMTP Set form_backend to php or omit it. The server sends e-mail using mail() or the smtp_* settings in includes/config.php. Prefer this when institute hosting already delivers mail reliably.
Sheet webhook Set form_backend to sheet. After validation, either PHP cURL POSTs JSON (secret) or, if curl_init is missing, the browser fetch() POSTs JSON (same sheet_webhook_secret as browser_token when Script BROWSER_TOKEN is unset). The script appends a Sheet row and uses MailApp for mail_to. Redeploy Apps Script from includes/google-sheet-webhook-apps-script.txt after updates.
Sheet (browser) form_backend = sheet_browser: always browser fetch() after validation (never PHP cURL). Optional sheet_browser_token + BROWSER_TOKEN; or omit sheet_browser_token and use sheet_webhook_secret only. See § Browser → Script.

Browser → Apps Script (no server cURL)

When curl_init is missing and form_backend is sheet, the site automatically uses the browser to fetch() JSON to the same Web App URL after honeypot, CSRF, and CAPTCHA pass — keep sheet_webhook_url and sheet_webhook_secret (12+ characters). The payload includes browser_token (same value as sheet_webhook_secret when sheet_browser_token is unset). Apps Script authorises it if property BROWSER_TOKEN is not set and browser_token matches WEBHOOK_SECRET (see includes/google-sheet-webhook-apps-script.txt).

form_backend sheet_browser does the same browser delivery even when PHP cURL exists. Optional sheet_browser_token matches Script BROWSER_TOKEN when you want a token separate from WEBHOOK_SECRET.

Trade-offs (vs sheet with PHP cURL)

  • JavaScript required on contact.php to complete delivery (small inline script). Users with JS disabled see a noscript notice.
  • Token in page: After validation, the base64 payload includes browser_token. Use a long random sheet_webhook_secret / WEBHOOK_SECRET and monitor the Sheet. Server-side cURL keeps the secret off the wire from the visitor’s DOM when you do not need the browser path.
  • CORS / network: If the browser blocks cross-origin POST to script.google.com, delivery fails; the UI suggests e-mail.

Default recommendation

Prefer sheet with PHP cURL when the institute enables it (secret stays server-side). Use automatic browser delivery (still form_backend sheet) or explicit sheet_browser when the host has no cURL or you require browser-only delivery.

Other alternatives if cURL is a problem

  • Keep form_backend sheet with sheet_webhook_url + sheet_webhook_secret; redeploy Apps Script from the repo.
  • Enable php-curl on the server if policy allows, for server-side POST.
  • form_backend php or google as other mail/embed options.
  • Server-side file_get_contents with a stream context could replace cURL in theory; not implemented in this repo.

2. Sheet mode — how the inbox and Reply-To are determined

When a visitor submits the form, PHP builds JSON that always includes mail_to taken from includes/config.php → 'contact'['mail_to'] (validated as an e-mail address). The Apps Script (see includes/google-sheet-webhook-apps-script.txt) uses this priority for the To: field of the notification:

  1. data.mail_to from the JSON (i.e. your config.php mail_to) — primary;
  2. if missing, the script property DESTINATION_EMAIL (optional fallback in Apps Script Project Settings);
  3. if still missing, the hard-coded default shms2026@mnnit.ac.in inside the script.

Reply-To. The script always sets replyTo to the visitor’s e-mail from the form. When secretariat staff click Reply in their mail client, the reply is addressed to the visitor, not to the webhook.

Subject line. The script builds: [SHMS-2026 Contact] + topic label + — + the visitor’s subject field.

Body. Plain text: name, e-mail, topic, role, urgency, subject, full message, then IP and user-agent for support diagnostics.

3. What the hosting administrator must set in includes/config.php (Sheet mode)

Create includes/config.php from includes/config.sample.php on the server (never commit secrets to a public repository). Inside 'contact' => array( … ) set at minimum:

'form_backend'         => 'sheet',
'sheet_webhook_url'    => 'https://script.google.com/macros/s/…/exec',
'sheet_webhook_secret' => '…same string as WEBHOOK_SECRET in Apps Script…',
'mail_to'              => 'shms2026@mnnit.ac.in',
  • mail_to — The conference secretariat address that should receive every notification. It is both shown in user-facing copy (where applicable) and embedded in the JSON for the script.
  • sheet_webhook_secret — Must match WEBHOOK_SECRET in Apps Script exactly (no spaces, same case).
  • sheet_webhook_url — The current Web app URL ending in /exec; redeploy a New version in Apps Script after code changes.

Optional (server-side only). sheet_webhook_cainfo — absolute path to a CA bundle (e.g. cacert.pem) if PHP reports SSL/cURL error 60 on the institute host. mail_from, mail_envelope_from, smtp_*, and mail_log_path apply to PHP mail mode, not to Sheet webhook delivery.

5. Operational checks before go-live

  • Sheet mode: Submit one test from https://www.mnnit.ac.in/…/contact.php (or the deployed path). Confirm (a) a new row on the Submissions tab and (b) one e-mail at mail_to with Reply-To set to the test address.

6. Google Form iframe mode (brief)

If form_backend is google and google_form_embed_url is set, the embedded form handles submission; PHP does not send contact mail for that path. E-mail for that option is configured in Google Forms / separate Apps Script—see includes/google-form-email-apps-script.txt in the repository if that mode is used.

How the Sheet approach is implemented (repository)

The browser never calls Google directly. The visitor posts to your PHP site; PHP validates the submission and, when form_backend is sheet, sends one server-side HTTPS POST (JSON) to your Apps Script Web app. The script checks a shared secret, appends a row, then sends mail.

End-to-end flow

Browser → POST contact.php (form fields + csrf + captcha) contact.php → shms_contact_handle_request() [includes/contact-form.php] → honeypot, CSRF, CAPTCHA, shms_contact_validate_and_collect() → shms_contact_dispatch_submission() → sheet: shms_contact_post_sheet_webhook() [cURL JSON POST] → php: shms_contact_send_mail() [SMTP / mail()] On success: redirect contact.php?sent=1#message-us Server (PHP) → HTTPS POST → https://script.google.com/macros/s/…/exec Apps Script doPost(e) → JSON.parse(postData), verify data.secret === WEBHOOK_SECRET → SpreadsheetApp.openById(SPREADSHEET_ID), appendRow(...) → MailApp.sendEmail({ to: mail_to, replyTo: visitor email, … }) → return JSON { "ok": true }

Success and errors

  • The Web app must respond with JSON containing "ok": true. PHP treats any other body, empty response, or cURL failure as an error and shows a message on the contact page (missing cURL, TLS/firewall, bad URL, secret mismatch, etc.).
  • With form_backend sheet, PHP smtp_* is not used for contact delivery; e-mail is sent by Google MailApp inside the script.

PHP mail() or SMTP (institute server)

Use this path when form_backend is php or omitted. All keys live under 'contact' in includes/config.php; see includes/config.sample.php for commented examples.

  1. Set mail_to to the secretariat inbox (e.g. shms2026@mnnit.ac.in).
  2. Optionally set mail_from and mail_envelope_from to addresses your mail infrastructure accepts (SPF/DKIM policies may apply).
  3. If the Computer Centre provides SMTP instead of local sendmail, set smtp_host, smtp_port (often 587 with TLS), smtp_encryption, smtp_user, smtp_pass, and optionally smtp_from / smtp_timeout.
  4. For diagnostics on a dedicated server, set mail_log_path to a writable path outside the web root.

On Windows development machines, mail() often has no MTA — use SMTP or test Sheet mode where curl_init is available. Production on www.mnnit.ac.in should follow the institute mail policy agreed with the organising team.

Google Sheet + webhook — reference setup (Sheet side often already done)

The steps below are the full checklist for creating the Sheet, Apps Script, and Web app from scratch. If the organising team has already completed this work, the institute hosting administrator need only align includes/config.php on the Linux server with the live /exec URL and secret, complete Part A (Apache / PHP), and verify e-mail using § E-mail messaging configuration. Each successful submission on contact.php still (1) adds a row in the Sheet and (2) triggers MailApp e-mail with Reply-To set to the visitor.

Before you start
  • A Google account with access to Google Drive and Google Sheets.
  • The SHMS site running on PHP with cURL enabled (see Step 6).
  • The script file in this project: includes/google-sheet-webhook-apps-script.txt (open it in a text editor so you can copy everything).

  1. Step 1 — Create the Google Sheet (log of responses)

    This Sheet is where each contact message will appear as a new row.

    1. Open Google Drive in your browser.
    2. Click New → Google Sheets → Blank spreadsheet.
    3. (Optional) Rename the file at the top (e.g. “SHMS-2026 contact responses”).
    4. Look at the browser address bar. The URL looks like:
      https://docs.google.com/spreadsheets/d/LONG_RANDOM_ID/edit
    5. Copy only the LONG_RANDOM_ID part (between /d/ and /edit). This is your SPREADSHEET_ID. Keep it in a safe note — you will paste it into Script properties in Step 3.

    Note. You do not need to create a tab named “Submissions” by hand. The script will create a worksheet called Submissions automatically the first time it runs, and it will write the column headers on the first row if the sheet is empty.

  2. Step 2 — Open Apps Script from that Sheet

    The code that receives data from your website lives in the same “project” as your Sheet.

    1. With your new Sheet open, click the menu Extensions → Apps Script.
    2. A new tab opens with the Apps Script editor. You may see sample code like function myFunction() { }.
    3. Select all text in the editor and delete it.
    4. Open includes/google-sheet-webhook-apps-script.txt from the SHMS website project on your computer. Select all, copy, paste into the empty Apps Script editor.
    5. Click Save (disk icon). Give the project a name if asked (e.g. “SHMS contact webhook”).

    First-time authorization. If Google asks you to authorize the script, choose your account and allow the permissions (Sheets, sending mail as you, etc.). This is normal for SpreadsheetApp and MailApp.

  3. Step 3 — Script properties (secret + Sheet ID)

    These values are not hard-coded in the pasted file; you add them in Project settings so you can change them without editing code.

    1. In Apps Script, click the gear icon (Project Settings) on the left.
    2. Scroll to Script properties → click Add script property for each row below.
    PropertyRequired?What to enter
    WEBHOOK_SECRETYesA long random password (at least 12 characters). Example style: mix letters and numbers. You will type the same value later in includes/config.php as sheet_webhook_secret. Anyone who knows both the Web app URL and this secret could post fake rows — treat it like a password.
    SPREADSHEET_IDYesThe ID you copied in Step 1 (from the Sheet URL). Not the “Script ID” shown elsewhere in settings — only the Sheet document ID.
    SHEET_NAMENoLeave unset to use the default tab name Submissions. If you set it, the script will create that tab if missing.

    After adding properties, click Save project if prompted. You do not need to add properties for every deployment — they stay with the script project.

  4. Step 4 — Deploy as a Web app (get the /exec URL)

    Deployment creates the HTTPS address your PHP server will call.

    1. Click the blue Deploy button (top right) → New deployment.
    2. Click the gear or “Select type” and choose Web app.
    3. Set Execute as: Me (your Google account).
    4. Set Who has access: Anyone. This is required: the PHP server is not logged into Google when it sends the POST.
    5. Click Deploy. You may need to authorize again.
    6. Copy the Web app URL shown in the dialog. It must end with /exec (production). Keep this URL private like an API endpoint.

    After you change the script code, use Deploy → Manage deployments → edit the deployment → Version “New version” → Deploy. Old URLs usually keep working, but the site must always use the current /exec link if Google shows a new one.

    Do not use a /dev URL for production; it is only for quick tests.

  5. Step 5 — Configure the SHMS website (includes/config.php)

    Open includes/config.php in your site and set at least these keys inside 'contact' => array( ... ):

    'form_backend'         => 'sheet',
'sheet_webhook_url'    => 'https://script.google.com/macros/s/YOUR_DEPLOYMENT_ID/exec',
'sheet_webhook_secret' => 'EXACT_SAME_STRING_AS_WEBHOOK_SECRET',
'mail_to'              => 'shms2026@mnnit.ac.in',
    • sheet_webhook_url — paste the full Web app URL from Step 4 (must be https, host script.google.com, path ending in /exec).
    • sheet_webhook_secret — must match WEBHOOK_SECRET in Apps Script exactly (same characters, no extra spaces).
    • mail_to — the inbox that receives the notification e-mail (and appears in the JSON for the script).

    Do not commit real secrets to a public Git repository. If the file is shared, rotate WEBHOOK_SECRET and sheet_webhook_secret together and redeploy.

  6. Step 6 — Enable PHP cURL and network access

    The contact handler uses PHP’s cURL extension to POST JSON to Google.

    1. Windows: Edit the active php.ini (path from php --ini or from opening tools/php-curl-check.php in the browser). Find the line ;extension=curl and remove the semicolon so it reads extension=curl. Save. Restart your web server or npm run dev.
    2. Linux server: Install the curl module for your PHP version (e.g. sudo apt install php-curl), then restart PHP-FPM or Apache.
    3. Verify: open tools/php-curl-check.php on the same host as the site (e.g. http://127.0.0.1:8080/tools/php-curl-check.php) until it says curl_init available: YES.

    Firewall & McAfee — allow outbound HTTPS to Google

    PHP must be allowed to open outbound TCP connections to port 443 (HTTPS) to Google hosts such as script.google.com. On a typical home PC, this already works; problems usually appear when a third-party firewall or “smart” antivirus blocks php.exe or unknown programs.

    Quick test (before changing rules)

    1. Open PowerShell or Command Prompt.
    2. Run: curl.exe -I https://script.google.com in PowerShell (PowerShell’s curl is an alias for Invoke-WebRequest, which will prompt for parameters — always use curl.exe). In Command Prompt (cmd), curl -I https://script.google.com is fine.
    3. You should see HTTP headers starting with HTTP/1.1 or HTTP/2 and a status such as 302 or 200. If this command hangs or says “Failed to connect” or “timeout”, the network path to Google is blocked (firewall, proxy, or offline).
    4. Optional: curl.exe -I https://www.google.com (PowerShell) — if Google works but only script fails, still check antivirus “web” rules for script URLs.

    Windows Defender Firewall

    1. Press Win + R, type wf.msc, Enter → Windows Defender Firewall with Advanced Security opens.
    2. Click Outbound Rules in the left tree. By default Windows allows outbound traffic unless a rule blocks it. Look for any rule that Blocks php.exe, your web server (httpd.exe, nginx.exe), or “All programs” on port 443 — if you find one, disable it or change to Allow (only if you understand why it was added).
    3. To explicitly allow PHP (recommended if you suspect blocking): Outbound Rules → New Rule… → Program → browse to your PHP executable (e.g. C:\Program Files\PHP\current\php.exe or your XAMPP php.exe) → Allow the connection → tick Domain, Private, Public as appropriate → name it e.g. “PHP outbound for SHMS”.
    4. Alternative path: Settings → Privacy & security → Windows Security → Firewall & network protection → Allow an app through firewall → Change settings → Allow another app… → add php.exe and enable Private (and Public if you use public Wi‑Fi).

    McAfee (Total Protection / LiveSafe / similar)

    Menus vary by version; look for Firewall, PC Security, or Web protection.

    1. Open the McAfee application from the system tray or Start menu.
    2. Find Firewall (sometimes under PC Security or Protect your PC). Open Settings or Firewall settings.
    3. Look for Program permissions, Internet connections for programs, or Ports and system services. Add or locate php.exe (full path to the same binary your site uses — check php --ini / Task Manager when npm run dev is running) and set it to Full / Allow / Outgoing access. If McAfee lists “PHP” or “Command line” generically, allow outbound.
    4. If there is Smart Access or Net Guard, temporarily set a less strict mode or add an exception for trusted local development — or add script.google.com to allowed destinations if your product offers a URL allow list (not all do).
    5. WebAdvisor / Secure VPN: These usually affect the browser, not server-side PHP. If only the contact form fails but curl.exe -I https://script.google.com from PowerShell works, the block is likely inside PHP/cURL or the web stack, not McAfee web scanning.
    6. Isolation test: Temporarily turn McAfee Firewall off for 1–2 minutes (McAfee → Firewall → turn off), submit the contact form again, then turn the firewall back on. If it works only while off, create a proper allow rule for php.exe instead of leaving the firewall disabled.

    SSL / cURL error 60 (Windows)

    If php tools/shms-webhook-cli-test.php shows cURL error 60, PHP cannot verify Google’s certificate (missing CA bundle). The repo includes tools/cacert.pem; the site uses it automatically for Sheet webhook calls. If the file is missing, download cacert.pem into tools/, or set curl.cainfo in php.ini to that file’s path, or optional sheet_webhook_cainfo in includes/config.php.

    Corporate / institute networks

    Some networks block direct HTTPS to the internet or require an HTTP proxy. PHP cURL must be configured for that proxy (curl_setopt CURLOPT_PROXY) — the SHMS project does not set a proxy by default. Ask IT whether outbound 443 to *.google.com is allowed from your PC or server.

  7. Step 7 — Send a test message and confirm you received it

    End-to-end test from the visitor’s point of view:

    1. Open the contact page: e.g. https://yoursite.example/contact.php or locally http://127.0.0.1:8080/contact.php when using npm run dev.
    2. Fill in every required field (name, e-mail, topic, urgency, subject, message).
    3. Answer the Security check (small maths question) correctly.
    4. Click Send message. If everything works, you should see the green Message sent banner after the page reloads.

    Where to look for the response (two places):

    1. Google Sheet → open your file → click the Submissions tab (bottom). A new row should appear with timestamp, names, e-mail, topic, role, urgency, subject, message, IP, and browser info.
    2. E-mail → the address in mail_to should receive a plain-text message summarising the enquiry. In Gmail (or similar), click Reply — it should address the visitor’s e-mail (Reply-To), not only the generic inbox.

    Column reference (first row is headers, then one row per submission):
